[01] 비전 제시                                  (0:00-0:20 / 20초)
"약관을 끝까지 읽고 동의하시는 분, 손 들어보세요. (잠시 멈춤)
거의 없으시죠. 대부분의 약관은 법률 용어로 가득해, 끝까지 읽고
이해하기 어렵게 설계되어 있습니다.

Guardian AI는 '읽지 않는 약관'과 '점점 정교해지는 피싱' 사이에서,
사용자 편에 서는 최후의 요새 — The Last Fortress입니다."

[02] 문제 정의                                  (0:20-0:40 / 20초)
"문제는 두 가지입니다. 첫째, 정보 비대칭 — 약관을 작성하는
기업은 알지만, 사용자는 의도적으로 이해하기 어렵게 설계된
법률 용어 때문에 그 내용을 알지 못한 채 동의합니다.
둘째, 늦은 대응 — 최근 티빙(TVING) 개인정보 유출 사고처럼,
국내 개인정보 침해 사건은 해마다 이어지고 있지만, 사용자는
피해가 발생한 뒤에야 그 사실을 알게 됩니다."

[03] 프로젝트 계획 (WBS & 기술 스택)             (0:40-1:00 / 20초)
"프로젝트는 5개 영역, 약 48일 분량의 WBS로 쪼갰습니다.
사용자 영역, 핵심 기능 영역(약관 분석·피싱 탐지), 데이터·
백엔드, 문서·발표 영역은 모두 100% 완료했고, 운영 영역도
75%까지 진행했습니다.

기술 스택은 Flutter + Dart, 상태관리는 setState, AI는
Anthropic Claude API를 http 패키지로 직접 호출합니다.
백엔드 서버는 없습니다 — 이것도 의도된 선택이고, 잠시 뒤
ADR로 설명드릴게요."

[05] 구현 방법 설명                              (1:10-1:30 / 20초)
"구조는 2개 레이어입니다. Presentation(화면, StatefulWidget)
→ Services(API 호출·로컬 분석·로컬 저장) → Claude API.
TosService는 약관을 4,000자 단위로 청킹해서 분석하고,
PhishingService는 로컬 규칙(키워드·단축 URL·의심 도메인)으로
1차 판정 후 Claude API로 2차 문맥 분석을 더하는 2단계
구조입니다. 의존 방향은 Presentation → Services 한쪽으로만
흐릅니다."

------------------------------------------------------------

[06] 활용 방안 및 기대 효과                      (1:30-1:45 / 15초)
"활용 시나리오는 간단합니다. 새 서비스 가입 전 약관을
붙여넣으면 위험 조항을 빨강/노랑/초록으로 분류해주고,
의심스러운 문자나 링크를 받으면 바로 피싱 여부를 확인할 수
있습니다. 사용자가 긴 약관을 직접 다 읽지 않아도 핵심 위험을
바로 파악하고, 스미싱 피해를 사전에 막는 게 목표입니다."

[07] ADR 요약 (질의응답 준비)                    (1:45-2:10 / 25초)
"핵심 결정 5가지를 ADR로 남겼습니다.
첫째, 모바일 프레임워크는 Flutter — 1인 개발 + 6주 안에
Android/iOS 동시 지원이 필요해서입니다.
둘째, 상태관리는 setState — Provider도 검토했지만, 화면 3개
규모에서는 화면 간 상태 공유 요구가 거의 없어 setState로
충분했습니다.
셋째, 백엔드 없음 — 사용자 데이터를 서버에 저장하지 않는다는
Privacy First 원칙과 일치하고, 6주 일정에서 서버 운영 비용을
제거합니다.
넷째, 인증 없음 — 로그인 없이 로컬에서만 동작하니 계정
시스템 자체가 공격 표면이 안 됩니다.
다섯째, 배포는 APK 사이드로드 — 스토어 심사 없이 발표 당일
즉시 설치해 시연할 수 있습니다."

[12] 구현 시행착오 사례                          (3:00-3:20 / 20초)
"개발 중 겪은 시행착오를 공유하면, 피싱 탐지 점수 로직에서
'점수 = 키워드수x2 + 도메인수x3' 식으로 계산했는데, 단축
URL 하나만 있어도 위험으로 떠야 하는 케이스가 safe로 나오는
버그가 있었습니다. 테스트를 먼저 돌려서 잡았고, 곱셈 합산
대신 '단축 URL이 있으면 무조건 danger' 같은 규칙 기반 분기로
바꿔서 해결했습니다. 이 디버깅 기록은 lessons 폴더에 누적해서,
같은 실수를 반복하지 않도록 관리합니다."
