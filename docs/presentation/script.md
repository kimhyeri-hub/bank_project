## 1:10 – 2:10 (60초) — 구현 방법 (아키텍처 + ADR)

> "구조는 2개 레이어입니다. Presentation(화면, StatefulWidget) → Services(API 호출·
> 로컬 분석·로컬 저장). Services가 Claude API를 직접 호출하고, 결과를 setState로
> 화면에 반영합니다. 화면이 3개(홈/약관/피싱)뿐인 MVP 규모라 의존 방향도 Presentation
> → Services 한쪽으로만 단순하게 흐릅니다.
>
> 핵심 결정 5가지를 ADR로 남겼습니다.
> 첫째, **모바일 프레임워크는 Flutter** — 1인 개발 + 6주 안에 Android/iOS 동시 지원이
> 필요해서입니다.
> 둘째, **상태관리는 setState** — Provider도 검토했지만, 화면 3개 규모에서는 화면 간
> 상태 공유 요구가 거의 없어 setState로 충분했고 학습 비용도 줄었습니다. 앱이 커지면
> Provider로 옮기는 마이그레이션 경로는 ADR에 남겨뒀습니다.
> 셋째, **백엔드 없음** — 사용자 데이터를 서버에 저장하지 않는다는 Privacy First 원칙과
> 일치하고, 6주 일정에서 서버 운영 비용을 제거합니다.
> 넷째, **인증 없음** — 로그인 없이 로컬에서만 동작하니 계정 시스템 자체가 공격 표면이
> 안 됩니다.
> 다섯째, **배포는 APK 사이드로드** — 스토어 심사 없이 발표 당일 즉시 설치해 시연할 수
> 있습니다.
>
> 약관 분석은 4,000자 단위로 청킹해서 API에 보내고, 피싱 탐지는 로컬 규칙(키워드·
> 단축 URL·의심 도메인)으로 1차 판정 후 Claude API로 2차 문맥 분석을 더하는 2단계
> 구조입니다."

- [x] ADR 최소 3개 표시 — **5개** 모두 구두로 언급 (질의응답 대비, 정확한 근거)

## 2:10 – 3:00 (50초) — 시행착오 + 성능/품질 + 테스트

> "개발 중 겪은 시행착오 하나를 공유하면, 피싱 탐지 점수 로직에서 '점수 = 키워드수×2 +

> `safe`로 나오는 버그가 있었습니다. 테스트를 먼저 돌려서 잡았고, 곱셈 합산 대신

> 이런 디버깅 기록은 `lessons/` 폴더에 누적해서, 같은 실수를 반복하지 않도록
