# Guardian AI — 질의응답 준비

> 발표 후 예상 질문 및 답변 요약. 외우기보다 논리 흐름을 이해하고 자기 말로 답하는 것을 권장합니다.

## 🤖 Claude Code / AI 활용

**Q. Claude Code를 어떻게 활용했나요?**
> 세 가지로 나눠요. 첫째, 반복 작업을 슬래시 커맨드로 표준화했어요 — `/analyze-tos`, `/check-phishing`, `/run-tests` 세 개를 `.claude/commands/`에 정의해서 명령 하나로 실행했어요. 둘째, `AGENTS.md` 한 파일에 AI 협업 규칙·ADR 위치·슬래시 커맨드 목록·작업 규칙을 다 모아서, AI가 새 세션을 시작해도 이 파일 하나만 읽으면 프로젝트 컨텍스트를 바로 파악할 수 있게 했어요. 셋째, 디버깅 교훈을 `lessons/`에 따로 쌓아서 같은 실수를 반복하지 않게 관리했어요.

**Q. AI가 코드를 다 짜준 거 아닌가요? 본인이 구현한 건가요?**
> 아키텍처 설계, ADR 의사결정, 테스트 케이스 설계는 제가 직접 했고요, AI는 보일러플레이트 구현과 리팩터링에서 속도를 높여주는 도구로 활용했어요. 예를 들어 피싱 탐지 점수 로직 버그는 제가 테스트를 먼저 짜고 실패 원인을 분석한 다음에 수정했어요. AI가 쓴 코드도 반드시 직접 검토하고 나서 커밋했습니다.

**Q. `lessons/` 폴더는 뭔가요?**
> 디버깅 교훈을 코드와 분리해서 쌓아두는 위키예요. 커밋 메시지에 묻어버리면 나중에 찾기 어렵고, AI도 다음 세션에서 같은 실수를 반복할 수 있어서요. 새 세션 시작할 때 AI가 `lessons/` 목록을 먼저 확인하게 규칙을 잡아뒀어요.

**Q. 슬래시 커맨드를 직접 만든 건가요?**
> 네, `.claude/commands/` 폴더에 마크다운 파일로 정의해요. 예를 들어 `/run-tests`는 전체 테스트를 실행하고 결과를 요약해주는 커맨드예요. 자주 쓰는 분석·검증 작업을 표준화해서 매번 같은 프롬프트를 반복 입력하지 않아도 됩니다.

## 🏗️ ADR / 기술 결정

**Q. Flutter 말고 React Native는 왜 안 썼나요?**
> 혼자 6주 안에 Android·iOS 동시 지원이 필요했는데, Flutter는 단일 Dart 코드베이스에 Hot Reload까지 돼서 개발 속도가 더 빨랐어요. React Native는 JS bridge 오버헤드랑 네이티브 모듈 설정 복잡도가 단기 일정엔 부담이었어요.

**Q. setState 말고 Riverpod이나 Bloc은요?**
> Provider도 검토했는데, 화면이 3개고 화면 간에 공유해야 할 상태가 거의 없어요. 그 규모에서 Riverpod·Bloc은 over-engineering이라 판단했고, setState로 충분했어요.

**Q. 백엔드가 없으면 데이터 신뢰성이 어떻게 되나요?**
> 이 앱은 사용자가 입력한 텍스트를 Claude API로 바로 분석해서 결과를 보여주는 도구예요. 서버에 데이터를 쌓을 필요가 없는 구조예요. 분석 이력은 기기 로컬(`shared_preferences`)에만 저장되고, 서버가 없으니 서버 해킹으로 대규모 개인정보가 유출될 위험도 없어요.

**Q. 로그인이 없으면 사용자 식별이 안 되는 거 아닌가요?**
> 이 앱은 개인화 서비스가 아니라 분석 도구예요. 로그인이 필요 없어요. 오히려 계정 DB 자체가 공격 표면이 되는데, 인증이 없으면 그 위험이 아예 사라져요.

**Q. APK 사이드로드는 보안 문제 아닌가요?**
> 교육 목적 프로젝트 배포고, 스토어 심사는 6주 일정 안에 맞출 수 없어요. 실제 서비스라면 Play Store 배포로 가야 하고, 그 과정(앱 서명·키 설정)은 WBS에 예정 항목으로 남겨뒀어요.

## 🔧 아키텍처 / 구현

**Q. Services가 Presentation을 참조하지 않는 이유가 뭔가요?**
> 단방향 의존성 원칙이에요. 비즈니스 로직이 UI에 종속되면 화면 없이 테스트하기 어렵고 재사용도 힘들어요. Services를 독립적으로 유지해야 `flutter test`로 화면 없이 로직만 검증할 수 있어요.

**Q. 4,000자 청킹 기준은 어떻게 정했나요?**
> Claude API 컨텍스트 한계와 응답 품질 사이 균형이에요. 너무 짧으면 조항 문맥이 끊기고, 너무 길면 API 제한에 걸리거나 응답이 느려져요. 4,000자는 일반적인 약관 한 조항~몇 개 분량이라 문맥 단절 없이 분석이 가능한 단위예요.

**Q. Claude API가 다운되면 피싱 탐지가 안 되나요?**
> 아니요, 로컬 1차 판정이 항상 먼저 돌아가요. Claude API는 2차 문맥 분석용이고, API 오류나 미설정 시엔 로컬 판정 결과로 자동 fallback해서 항상 판정 결과가 나와요.

**Q. 개인정보는 어떻게 처리되나요?**
> 서버엔 아무것도 안 저장돼요. Claude API 호출 시 텍스트가 전송되긴 하지만, 분석 이후 서버에 남지 않아요. 분석 이력은 기기에만 로컬 저장되고요.

**Q. 피싱 탐지 정확도는 어느 정도인가요?**
> 완벽한 탐지는 불가능해요. 로컬 키워드·단축 URL·의심 도메인 기반 1차 판정에 Claude API 2차 문맥 분석을 더해 보완하는 구조예요. 실제 서비스라면 공신력 있는 피싱 DB와 연동해 정확도를 높여야 하고, 향후 계획에 포함되어 있어요.

**Q. iOS는 지원하나요?**
> Flutter로 개발했으니 코드는 iOS 호환이에요. 빌드는 macOS + Xcode 환경이 필요해서 이번 발표는 Android APK로 시연합니다.

## 💡 한 줄 요약 (외우기용)

| 주제 | 핵심 한 마디 |
|------|------------|
| Flutter 선택 | 1인 개발 + 6주 + Android/iOS 동시 → 단일 코드베이스 필요 |
| setState | 화면 3개, 화면 간 공유 상태 거의 없음 → over-engineering 피함 |
| 백엔드 없음 | 분석 도구라 서버 저장 불필요 + Privacy First |
| 인증 없음 | 개인화 불필요 + 계정 DB 자체가 공격 표면 |
| APK 사이드로드 | 교육 목적 + 스토어 심사 일정 불가 |
| Claude Code | AGENTS.md 통합 + 슬래시 커맨드 + lessons/ 위키 |
| 단방향 의존성 | Services 독립 → UI 없이 테스트 가능 |
| 4,000자 청킹 | 문맥 단절 없이 API 제한 회피하는 최적 단위 |
| API fallback | 로컬 1차 판정 항상 실행 → API 없어도 결과 보장 |
