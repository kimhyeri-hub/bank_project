01 · 비전 제시

🎓 캡스톤 디자인 · Flutter · Claude API

Guardian AI
The Last Fortress

당신의 프라이버시를 지키는 최후의 요새

"복잡한 약관과 거대 AI의 파도 속에서,
당신의 소중한 프라이버시를 지켜내는 최후의 요새"

모든 사람이 약관을 이해할 권리가 있다.
모든 사람이 자신의 데이터를 통제할 권리가 있다.
Guardian AI는 그 권리를 기술로 현실화한다.

🔐

Privacy First

모든 분석은 온디바이스(On-Device) 처리를 원칙으로 한다. Guardian AI 자신이 또 하나의 감시자가 되지 않는다.

⚡

Clarity Over Complexity

복잡한 약관을 짧고 명확하게. 법률 언어를 인간의 언어로. 모든 정보는 누구나 이해할 수 있는 형태로 제공한다.

🛡️

Proactive Defense

위협이 오기를 기다리지 않는다. 실시간으로 감지하고, 사전에 경고하며, 즉각 차단한다.

🤝

User Empowerment

단순히 정보를 제공하는 것을 넘어, 사용자 스스로 올바른 판단을 내릴 수 있도록 힘을 부여한다.

02 · 문제 정의

디지털 프라이버시 위기,
지금 당장 해결해야 한다

최근 티빙(TVING) 개인정보 유출 사고처럼, 국내 개인정보 침해 사건은 해마다 이어지고 있습니다. 기업은 약관에 모든 것을 적어두지만 정작 사용자는 그 내용을 알지 못한 채 동의하고, 피해는 늘 사고가 알려진 뒤에야 드러납니다.

⚖️

정보 비대칭

기업은 알지만 사용자는 모른다. 법률 용어로 가득한 약관은 의도적으로 이해하기 어렵게 설계된다.

🧠

인지 과부하

평균 수십 페이지에 달하는 약관. 하루에도 여러 번 반복되는 "동의하기" 버튼. 인간의 처리 한계를 넘어선다.

🎣

실시간 피싱 위협

AI로 생성된 정교한 피싱 메시지와 스미싱. URL 하나 클릭으로 모든 개인정보가 유출된다.

📵

늦은 대응

피해가 발생한 뒤에야 알게 되는 구조. 클릭하기 전, 가입하기 전에 위험을 알 방법이 없다.

Presentation → Services → Claude API (Services는 Presentation을 참조하지 않음 · 역방향 참조 금지)

06 · 활용 방안 및 기대 효과

복잡한 약관은 한눈에, 스미싱은 사전에

새 서비스에 가입하기 전, 의심스러운 문자를 받았을 때 — Guardian AI를 한 번 거치는 것만으로 판단이 바뀝니다.

📝

새 서비스 가입 전

약관 텍스트를 붙여넣거나 PDF를 업로드하면, 위험 조항을 🔴위험 / 🟡주의 / 🟢안전 3단계로 즉시 분류해 보여준다.

📩

의심스러운 문자·링크 수신 시

URL이나 문자 내용을 붙여넣으면 로컬 1차 판정 + Claude API 2차 분석으로 피싱 여부를 즉시 확인할 수 있다.

1명의 사용자가 Guardian AI를 사용할 때마다
긴 약관도 한눈에 정리 · 스미싱 피해 사전 탐지 및 방어

페르소나 설명 핵심 니즈
🧑‍💼 직장인 A 업무 중 다양한 SaaS 서비스에 가입 약관 빠른 요약, 위험 조항 알림
👩‍🎓 대학생 B 앱 설치 시 약관을 무조건 동의 쉬운 이해, 피싱 링크 탐지
👴 시니어 C 스미싱·보이스피싱 피해 취약계층 직관적 UI, 위협 즉시 경고
👩‍💻 개발자 D API·오픈소스 라이선스 검토 필요 상세 분석, 위험 점수 제공

07 · ADR 요약 (질의응답 준비)

핵심 아키텍처 결정 5가지

주요 설계 결정을 ADR(Architecture Decision Record)로 기록했습니다. 전체 내용은 .planning/decisions/ 참고.

ADR-0001
모바일 플랫폼: Flutter
1인 개발 + 6주 일정에서 Android/iOS를 동시에 지원하기 위해 단일 코드베이스를 선택.
ADR-0002
상태 관리: setState (Provider는 검토 후 보류)
화면 3개(홈/약관/피싱) 규모의 MVP에서는 화면 간 상태 공유 요구가 거의 없어 setState로 충분. 앱 규모가 커지면 Provider 마이그레이션 경로를 따름.
ADR-0003
백엔드 없이 Flutter + Claude API 직접 호출
사용자 데이터를 서버에 저장하지 않는다는 Privacy First 원칙과 일치. 6주 일정에서 서버 구축·운영 비용 제거.
ADR-0004
인증 없음 (로컬 전용)
로그인 없이 기기 내에서만 동작. 계정 시스템 자체가 공격 표면이 되지 않음.
ADR-0005
배포: APK 사이드로드
스토어 심사 없이 발표 당일 즉시 설치해 시연 가능.

08 · 앱 구조 설명

Presentation + Services 2계층

화면(Presentation)과 API 호출·로컬 분석·저장(Services)을 분리한 2계층 구조. 도메인/데이터 레이어를 별도로 두지 않아 MVP 규모에 맞는 단순한 구조를 유지합니다.

Presentation — lib/presentation/screens/

화면역할
home_screen.dart홈 — 활동 요약 대시보드, 다른 화면 진입점 (하단 탭)
tos_screen.dart약관 분석 — PDF/URL/텍스트 입력 (하단 탭)
phishing_screen.dart피싱 탐지 — URL·문자·이메일 입력 (하단 탭)
notifications_screen.dart공지사항 목록
history_screen.dart분석 이력 목록

Services — lib/services/

서비스역할
claude_service.dartClaude API 공통 HTTP 클라이언트 (complete, isConfigured)
tos_service.dart약관 분석 — 4,000자 청킹 + 프롬프트 + JSON 파싱 + mock
phishing_service.dart피싱 1차(로컬 키워드/URL) + 2차(Claude) 판정
document_service.dartPDF 바이트 → 텍스트 추출 (syncfusion_flutter_pdf)
web_file_picker*.dart웹 PDF 파일 선택 (조건부 export, 비웹 스텁 포함)
history_service.dart분석 이력 저장/조회 (shared_preferences, 최대 20건)
activity_service.dart사용 통계 (분석/검사 횟수)
notification_service.dart공지사항 데이터 및 읽음 상태
의존 방향: Presentation → Services → Claude API - Presentation은 필요한 Service를 직접 생성/호출 (setState로 결과 반영) - Services는 Presentation을 참조하지 않음 (역방향 금지) - Services 간 단방향 참조는 허용 (예: tos_service → claude_service)

12 · 구현 시행착오 사례

피싱 탐지 점수 로직 vs 테스트 기대값 불일치

디버깅·시행착오는 코드와 분리해 lessons/에 누적 관리합니다. (예시: lessons/01-phishing-score-test-mismatch.md)

🔴 증상

flutter test 실행 시 단위 테스트 2개 실패 — '확인하세요 http://bit.ly/win123'이 danger가 아닌 safe로, '계좌 잔액을 확인하세요.'가 warning이 아닌 safe로 판정됨.

🟡 원인 분석

점수 기반 로직의 임계값 문제. score = 키워드수×2 + 도메인수×3 방식에서 bit.ly 1개 = 3점, 계좌 1개 = 2점으로 모두 임계값(4점) 미달. 단축 URL처럼 단독으로 위험한 신호를 곱셈·합산만으로는 반영하지 못했다.

final score = (foundKeywords.length * 2 + foundDomains.length * 3).clamp(0, 10); final level = score >= 7 ? danger : score >= 4 ? warning : safe;

🟢 해결

숫자 연산 대신 규칙 기반 분기로 변경 — 키워드 2개 이상이거나 의심 도메인이 하나라도 있으면 무조건 danger로 분류.

if (foundKeywords.length >= 2 || foundDomains.isNotEmpty) { level = PhishingLevel.danger; } else if (foundKeywords.isNotEmpty) { level = PhishingLevel.warning; } else { level = PhishingLevel.safe; }
🧪

교훈 1

로직을 이관할 때 동작 검증을 먼저 한다. 테스트를 먼저 작성했다면 곧바로 잡을 수 있었다.

🎯

교훈 2

단독 위험 요소는 점수가 낮더라도 최소 레벨을 보장해야 한다. 단축 URL은 그 자체로 의심 신호다.

📣

교훈 3

테스트 실패 메시지를 믿어라. safe vs danger처럼 큰 차이는 로직 자체를 의심해야 한다.

13 · 성능 최적화 노력

응답 시간과 안정성을 위한 설계

Claude API 호출 비용·응답 시간과 사용자 대기 시간을 줄이기 위한 3가지 장치입니다.

✂️

4,000자 단위 청킹

긴 약관 텍스트를 TosService에서 4,000자 단위로 분할해 순차 호출. 토큰 제한을 넘기지 않으면서 청크별 결과를 누적 병합해 하나의 TosReport로 합친다.

⚡

로컬 1차 판정 우선

피싱 탐지는 로컬 규칙(키워드·단축 URL·의심 도메인) 판정을 먼저 수행해 즉시 결과를 제공하고, Claude API로 2차 문맥 분석을 보강한다.

🛟

API 미설정/오류 시 즉시 fallback

ClaudeService.isConfigured == false이거나 호출이 실패하면 대기 없이 로컬 분석(PhishingService._analyzeLocally, TosService.mockReport) 결과를 표시한다.

💾

로컬 저장 용량 관리

분석 이력은 shared_preferences에 최근 20건까지만 유지해 저장 공간과 조회 속도를 일정하게 유지한다.
