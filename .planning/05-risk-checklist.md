### R1: Claude API 토큰 비용 초과 / 응답 지연

- **카테고리**: 기술 / AI 의존
- **발생 가능성**: 중
- **영향도**: 상
- **배경**: 약관 분석(F-TOS-001)은 수천 토큰의 긴 입력이 반복 호출될 수 있고, PII 보조 인식과 피싱 탐지까지 합산 시 무료/저비용 한도를 초과할 위험이 있다. 응답 지연은 "30초 이내 분석" 비기능 요건에도 직접 영향을 준다.
- **대응**:
  - **예방**:
    - 일일 API 호출 횟수 상한 설정 (환경변수로 관리)
    - 동일 입력 해시 기반 응답 캐싱 적용 (SQLite 또는 메모리 캐시)
    - PII 탐지는 정규식 로컬 처리를 1차로, Claude API는 정규식 미탐 케이스 보조에만 사용
    - 약관 청킹 전략 사전 구현 → 긴 텍스트를 분할하여 호출 횟수 예측 가능하게 유지
  - **발생 시**:
    - 비용 한도 초과 시 Claude API 호출 차단 후 로컬 처리 결과만 반환 (기능 저하 모드)
    - 응답 30초 초과 시 타임아웃 처리 + 사용자에게 "분석 중 문제가 발생했습니다. 텍스트를 줄여 다시 시도해 주세요" 안내
- **모니터링 지표**:
  - 주별 API 호출 횟수 및 토큰 사용량 로그 확인 (Anthropic 콘솔)
  - 앱 내 API 호출 실패율 로그 (Flutter 디버그 콘솔)

### R2: 긴 약관 텍스트의 컨텍스트 한계 초과

- **카테고리**: 기술
- **발생 가능성**: 상
- **영향도**: 상
- **배경**: 실제 서비스 약관(카카오, 네이버 등)은 수만 자에 달하며, Claude API의 단일 요청 컨텍스트 한계를 초과할 가능성이 높다. 이는 약관 분석(F-TOS-001)의 핵심 기능을 무력화시킬 수 있다.
- **대응**:
  - **예방**:
    - 청킹(Chunking) 모듈을 약관 서비스 클래스(`tos_service.dart`) 설계 초기부터 포함
    - 청크 단위: 단락/조항 경계 기준 분할 → 각 청크별 위험 조항 추출 → 결과 병합
    - UI에서 입력 텍스트 길이 실시간 표시 및 최대 입력 권장 길이 사전 안내 (예: "10,000자 이하 권장")
    - PDF 업로드(F-TOS-002) 구현 시 텍스트 추출 단계에서 길이 제한 적용
  - **발생 시**:
    - 청크 분할 자동 처리 후 부분 결과 순차 표시 (진행률 UI 제공)
    - 분할 처리 실패 시 "텍스트가 너무 깁니다. 주요 조항만 붙여넣어 주세요" 안내
- **모니터링 지표**:
  - 12주차 실제 약관(카카오·네이버) 붙여넣기 테스트 시 오류 발생 여부 확인
  - 청크 분할 후 결과 병합 정확도 수동 검토

### R3: 피싱 탐지 오탐률 목표(20% 이하) 미달

- **카테고리**: 기술
- **발생 가능성**: 중
- **영향도**: 중
- **배경**: 정상 URL을 위험으로 잘못 판정하면 사용자 신뢰를 잃고 앱의 핵심 가치를 훼손한다. 특히 네이버·카카오 등 국내 주요 도메인의 단축 URL이나 마케팅 링크가 오탐될 가능성이 있다.
- **대응**:
  - **예방**:
    - 신뢰 도메인 화이트리스트 사전 구축 (naver.com, kakao.com, google.com 등 주요 도메인)
    - 위험 점수 산출 시 화이트리스트 도메인은 감점 처리
    - 외부 위협 DB(VirusTotal 무료 API) 연동 시 일일 한도 내에서만 사용, 초과 시 자체 패턴 매칭으로 대체
  - **발생 시**:
    - 13주차 검증 단계에서 정상 URL 10건 테스트 → 오탐률 측정 → 20% 초과 시 화이트리스트 확장 및 점수 가중치 재조정
    - 최종 발표까지 오탐률 미달 시 "참고용 위험 지표" 수준으로 UI 문구 조정 (확정 판정 대신 "주의 필요" 표현 사용)
- **모니터링 지표**:
  - 정상 URL 10건 / 의심 URL 10건 테스트 셋 정의 및 주차별 오탐률 기록
  - 13주차 검증 체크리스트: "정상 URL 오탐률 20% 이하 충족 여부"

- **카테고리**: 기술 / 일정
- **발생 가능성**: 중
- **영향도**: 상

    - UI 폴리싱 생략 후 기능 동작 우선 확보
- **모니터링 지표**:
  - 10주차 말: `flutter run` 실행 성공 여부

### R5: Android / iOS 플랫폼 권한 처리 이슈

- **카테고리**: 기술
- **발생 가능성**: 중
- **영향도**: 중
- **배경**: 클립보드 자동 감지(F-CLIP-001)는 Android foreground service 권한 이슈가 요구사항 문서에 명시되어 있다. 또한 파일 업로드(PDF), URL 공유 연동 등도 플랫폼별 권한 처리가 필요하다.
- **대응**:
  - **예방**:
    - 클립보드 자동 감지(F-CLIP-001)는 Should Have로 분류된 만큼, 권한 이슈 발생 시 MVP에서 제외하고 수동 붙여넣기 방식으로 대체
    - Android 권한은 `permission_handler` 패키지로 통일 처리
    - iOS와 Android 각각 `Info.plist` / `AndroidManifest.xml` 권한 선언 10주차 환경 구축 시 미리 작성
  - **발생 시**:
    - 권한 거부 시 사용자 친화적 안내 다이얼로그 표시 후 수동 입력 대체 흐름으로 전환
    - iOS 권한 이슈로 빌드 실패 시 Android 단일 플랫폼 시연으로 발표 대응 (요구사항에 Android 실기기 명시)
- **모니터링 지표**:
  - 11주차: 파일 선택 / 클립보드 접근 권한 동작 확인 (Android 에뮬레이터 + 실기기)
  - 14주차: iOS 시뮬레이터 빌드 성공 여부 확인

    - PII 마스킹은 정규식 기반 로컬 처리가 1차 완성 목표 → Claude API 보조는 선택
    - 중간 발표 시연 시나리오(`midterm_demo_script.md`)를 11주차 초에 먼저 작성 → 시연 범위를 확정한 뒤 개발
  - **발생 시**:
    - AI 보조 PII 인식 제외하고 정규식 기반 마스킹만으로 시연 (전화번호·이메일 마스킹은 정규식으로 충분)
    - 미완성 기능은 슬라이드에서 "개발 예정" 로드맵으로 제시
- **모니터링 지표**:
  - 11주차 수요일: 전화번호·이메일 마스킹 동작 여부 중간 점검
  - 11주차 금요일: 시연 시나리오 5분 리허설 완료 여부

### R7: 시험 기간·타 과목 과제와 일정 충돌

- **카테고리**: 일정
- **발생 가능성**: 상
- **영향도**: 중
- **배경**: 10~15주차는 학기 말 집중 기간으로, 기말시험(13~14주차 추정)·타 과목 보고서 마감이 겹칠 가능성이 높다. 1인 프로젝트 특성상 일정 지연 시 분담이 불가능하다.
- **대응**:
  - **예방**:
    - 10주차 시작 시 학기 전체 시험·과제 일정을 캘린더에 등록하고 WBS 일정과 충돌 여부 사전 확인
    - 충돌이 예상되는 주는 해당 주 작업량을 전주로 앞당겨 배분 ("버퍼 선사용")
    - WBS 합계에 이미 버퍼 20%가 포함되어 있음 → 충돌 주에는 해당 버퍼를 의도적으로 소진
  - **발생 시**:
    - Should Have 기능(히스토리, 대시보드, PDF 업로드) 전량 포기 후 Must Have 3개 기능 완성에만 집중
    - 해당 주 작업 목표를 "동작하는 기능 유지"로 하향 조정
- **모니터링 지표**:
  - 매주 일요일 주간 리뷰 시 다음 주 외부 일정 충돌 여부 1차 확인
  - 13~14주차 시험 일정 사전 파악 및 12주차 말 작업 완충 계획 수립

### R8: AI가 생성한 코드를 본인이 이해 못해 Q&A 실패

- **카테고리**: AI 의존
- **발생 가능성**: 중
- **영향도**: 상
- **배경**: 졸업 작품 발표에서는 코드 작동 원리, 구조적 선택 이유, 알고리즘 설명에 대한 심사위원 질문이 필수적으로 나온다. AI 보조로 작성된 코드를 이해 없이 사용하면 Q&A에서 치명적인 감점이 발생한다.
- **대응**:
  - **예방**:
    - AI가 생성한 코드는 반드시 줄 단위로 읽고 주석을 직접 작성한 뒤 커밋 (주석 없는 코드는 미이해 코드로 간주)
    - 핵심 알고리즘(PII 정규식 패턴, 청킹 로직, 위험 점수 산출)은 직접 재작성하거나 최소 1회 이상 설명 연습
    - AI에게 "이 코드를 5줄로 설명해줘"를 요청 후 본인이 그 설명을 자기 언어로 재서술하는 습관 유지
  - **발생 시**:
    - 이해 못 한 코드 발견 시 즉시 AI에게 설명 요청 → 이해 완료 후 진행
    - 발표 전날 "내가 설명할 수 없는 코드 목록" 작성 후 집중 복습
- **모니터링 지표**:
  - 각 기능 완성 시 "이 기능의 핵심 로직을 말로 3분 설명할 수 있는가" 자가 점검
  - 14주차 사용성 테스트 후 발표 Q&A 예상 질문 10개 직접 작성 및 답변 준비

### R9: AI 도구(Claude.ai) 자체 다운타임 / 사용량 한도 소진

- **카테고리**: AI 의존
- **발생 가능성**: 하
- **영향도**: 중
- **배경**: 개발 보조 도구로 Claude.ai를 활용하는 경우, 서비스 다운타임이나 무료/플랜 사용량 소진 시 개발 속도가 급격히 저하될 수 있다.
- **대응**:
  - **예방**:
    - Claude.ai 외 ChatGPT, Gemini 등 대체 AI 도구 계정 사전 준비
    - 핵심 프롬프트(약관 분석용, 피싱 분석용)는 로컬 파일(`prompts/`)에 저장 → AI 없이도 프롬프트 재사용 가능
  - **발생 시**:
    - 대체 AI 도구로 즉시 전환
    - 다운타임 중에는 문서 작업, 테스트 케이스 작성 등 AI 불필요 작업으로 전환
- **모니터링 지표**:
  - 주요 프롬프트 파일 `prompts/` 디렉토리 존재 여부 (10주차 중 생성)

### R10: 데모 당일 앱 동작 불가 (빌드 실패 / 환경 문제)

- **카테고리**: 결과물
- **발생 가능성**: 중
- **영향도**: 상
- **배경**: 발표 당일 빔프로젝터 연결, 네트워크 환경 변화, 실기기 배터리·OS 업데이트 등 통제 불가능한 환경 요인으로 앱 시연이 실패할 수 있다. Claude API 호출이 필요한 기능은 Wi-Fi 없이는 동작 불가하다.
- **대응**:
  - **예방**:
    - 14주차 Release APK를 실기기에 미리 설치하고 발표 당일 재설치 없이 사용
    - 발표 전날 모든 기능 동작 최종 확인 (온보딩 → PII → 약관 → 피싱 전체 흐름)
    - 핫스팟 데이터를 개인 스마트폰에 준비 (학교 Wi-Fi 불안정 대비)
    - 데모 시나리오의 입력 텍스트(약관 샘플, 스미싱 문자 샘플)를 앱 내 "예시 불러오기" 버튼으로 미리 탑재
  - **발생 시**:
    - 사전 녹화 데모 영상(`docs/demo_video.mp4`)으로 대체 시연 (15주차 산출물에 선택 항목으로 포함)
    - 실기기 오류 시 에뮬레이터 화면 미러링으로 즉시 전환
- **모니터링 지표**:
  - 14주차 금요일: Release APK 실기기 설치 + 전체 기능 동작 확인 완료 여부
  - 15주차 발표 전날: 데모 시나리오 리허설 완료 + 예비 영상 준비 여부

### R11: Claude API Anthropic 정책 변경 (모델 deprecated 등)

- **카테고리**: 기술 / AI 의존
- **발생 가능성**: 하
- **영향도**: 상
- **배경**: Guardian AI의 핵심 기능 3개(PII 보조, 약관 분석, 피싱 탐지) 모두 Claude API에 의존한다. Anthropic의 모델 정책 변경, 가격 정책 변경, 또는 특정 모델의 지원 종료 시 서비스 전체에 영향을 준다.
- **대응**:
  - **예방**:
    - API 클라이언트(`lib/services/`) 설계 시 모델명을 상수(환경변수)로 분리 → 모델 교체 시 1줄 수정으로 대응
    - 각 기능의 로컬 처리 폴백(정규식 PII, 패턴 매칭 피싱)을 항상 병행 구현
    - Anthropic 공식 변경 공지 채널(뉴스레터, 상태 페이지) 구독
  - **발생 시**:
    - 신규 모델명으로 상수 교체 후 프롬프트 재테스트 (통상 하루 이내)
    - API 완전 중단 시 로컬 처리 기능만으로 MVP 시연 (PII 정규식, 피싱 패턴 매칭)
- **모니터링 지표**:
  - 주차별 API 호출 성공률 확인
  - Anthropic 상태 페이지(status.anthropic.com) 발표 전날 확인
