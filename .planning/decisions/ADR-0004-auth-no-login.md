# ADR-0004: 인증 방식 선택 — 로그인 없는 로컬 전용 앱

- 상태: Accepted
- 날짜: 2026-05-19
- 결정자: Guardian AI 개발자

## 배경

앱에서 사용자 인증(로그인/회원가입) 기능을 구현할지 결정해야 한다.  
인증 유무에 따라 데이터 저장 방식, 개인정보 처리 방침, 개발 복잡도가 크게 달라진다.  
요구사항 문서(01-requirements.md)에는 "사용자 계정/로그인 시스템"이 **Won't Have**로 명시되어 있다.

## 고려한 대안

### 대안 A: 인증 없음 — 로컬 전용 (익명 앱)
- 장점:
  - 회원가입·로그인 UI 개발 불필요
  - 백엔드 인증 서버 불필요
  - 사용자 계정 정보 수집 없음 → 개인정보 처리 방침 최소화
  - 앱 진입 장벽 없음 → 즉시 사용 가능
  - Privacy First 원칙에 가장 부합
- 단점:
  - 기기 교체 시 분석 히스토리 유실
  - 사용자 식별 불가 → 개인화 기능 불가

### 대안 B: 이메일/비밀번호 로그인 (자체 구현)
- 장점:
  - 기기 간 데이터 동기화 가능
  - 사용자 식별 → 향후 개인화 기능 기반
- 단점:
  - 인증 서버 개발·운영 필요 → 백엔드 없는 ADR-0003 결정과 충돌
  - 비밀번호 해싱, 토큰 관리 등 보안 구현 복잡
  - 6주 일정 내 구현 불가능 (요구사항에 Won't Have 명시)

### 대안 C: Firebase Authentication (소셜 로그인)
- 장점:
  - Google/Apple 로그인 SDK로 구현 빠름
  - 토큰 관리 Firebase 위임
- 단점:
  - Firebase 연동 추가 (ADR-0003에서 Firebase 미채택)
  - 소셜 계정 연동 시 사용자 개인정보(이메일, 이름) 수집 → Privacy First 원칙 상충
  - Apple 로그인은 iOS App Store 배포 시에만 필수 (발표용 APK에는 불필요)

### 대안 D: 익명 인증 (Firebase Anonymous Auth)
- 장점:
  - 로그인 없이 서버 측 식별자 부여
  - 추후 계정 연동 업그레이드 가능
- 단점:
  - Firebase 의존성 추가 (ADR-0003 결정과 충돌)
  - 익명 UID 만료 시 데이터 유실 — 로컬 저장과 실질적 차이 없음
  - 구현 복잡도 대비 MVP에서 얻는 이점 없음

## 결정

**대안 A: 인증 없는 로컬 전용 앱**을 선택한다.

## 이유

- 요구사항 문서에 "사용자 계정/로그인 시스템 — 백엔드 인프라 필요, 6주 내 불가"로 Won't Have 명시
- Guardian AI의 핵심 가치 "Privacy First"는 사용자 식별 정보를 수집하지 않는 것을 포함한다. 로그인 없는 구조가 이 가치를 가장 순수하게 구현
- 분석 히스토리는 로컬 SQLite에 저장하면 인증 없이도 기기 내 이력 조회 가능 (F-HIST-001 충족)
- 발표 심사 기준에 인증 기능이 포함되지 않음

## 결과 (예상되는 영향)

긍정:
- 앱 실행 즉시 핵심 기능 사용 가능 → 온보딩 3단계 후 바로 진입
- 개인정보 처리 방침 작성 범위 최소화 (API 키 제외 사용자 개인정보 미수집)
- 인증 관련 보안 취약점(토큰 탈취, 세션 하이재킹 등) 원천 차단

부정 / 제약:
- 기기 교체·앱 재설치 시 로컬 히스토리 유실 → 온보딩에서 사전 안내
- 사용자 식별 불가 → 추후 클라우드 동기화, 개인화 기능은 MVP 이후 단계에서 별도 ADR 작성

## 후속 작업

- [ ] 온보딩 화면 3단계에 "로컬 저장 안내 — 히스토리는 이 기기에만 저장됩니다" 문구 추가
- [ ] 앱 재설치 시 SQLite 데이터 유실 동작 확인 (테스트 케이스 추가)
- [ ] 개인정보 처리 방침 최소 범위로 작성 (MVP에서는 생략 가능)
